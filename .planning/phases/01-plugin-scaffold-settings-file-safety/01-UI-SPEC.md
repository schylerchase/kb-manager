# Phase 1 — UI Design Contract

> Visual and interaction contract for the Plugin Scaffold + Settings + File Safety phase.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Context: This is an Obsidian plugin, not a web app. All UI is rendered inside the
> Obsidian settings panel via the PluginSettingTab API. Typography, color, and spacing
> are governed by Obsidian's CSS variable system and the user's active theme. The plugin
> does not ship its own design system — it inherits Obsidian's.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Obsidian Plugin API (native) |
| Preset | Not applicable |
| Component library | Obsidian Setting API — `new Setting(containerEl)`, `SliderComponent`, `ToggleComponent`, `TextAreaComponent`, `TextComponent`, `ButtonComponent` |
| Icon library | Obsidian built-in icon set (Lucide subset) — `layout-list` for ribbon icon |
| Font | Inherits from user's Obsidian theme via `--font-text` / `--font-ui` CSS variables |

---

## Spacing Scale

Obsidian uses a 4px base grid. The plugin must not override these values — use them as
alignment targets when writing any custom CSS in `styles.css`.

| Token | Value | Usage in settings tab |
|-------|-------|-----------------------|
| xs | 4px | Gap between toggle and its label; icon button padding |
| sm | 8px | Spacing between Setting description and control |
| md | 16px | Vertical gap between Setting rows (Obsidian default) |
| lg | 24px | Vertical gap after a section heading (`containerEl.createEl('h3', ...)`) |
| xl | 32px | Top padding of settings tab container |
| 2xl | 48px | Not used in Phase 1 |
| 3xl | 64px | Not used in Phase 1 |

Exceptions: Obsidian's `Setting` component self-manages its internal padding via
`--setting-item-padding` (typically 12px vertical, 0 horizontal). Do not override this
in `styles.css` for Phase 1.

---

## Typography

Phase 1 settings UI inherits all typography from Obsidian's CSS variables. Do not
declare custom font sizes. Reference these variables exactly:

| Role | Obsidian CSS Variable | Typical Rendered Size | Usage |
|------|-----------------------|-----------------------|-------|
| Body / setting description | `--font-ui-small` | ~13px | `.setting-item-description`, help text, placeholder text |
| Setting name / label | `--font-ui-medium` | ~14px (default) | `.setting-item-name`, section field labels |
| Section heading | `--font-ui-larger` | ~16px | `containerEl.createEl('h3', ...)` section headers |
| Monospace (delimiter examples) | `--font-monospace` | ~13px | Inline `<code>` elements in descriptions |

Line-height: defer to Obsidian's stylesheet (`--line-height-tight` ≈ 1.3 for headings,
`--line-height-normal` ≈ 1.5 for body/description). Do not set `line-height` in plugin CSS.

Font weights: Use Obsidian's defaults. Section headings render at `--font-semibold` (600),
body at `--font-normal` (400). Do not declare weight overrides.

## Color

All color values are Obsidian CSS variables. Do not use hex codes. The plugin's settings
tab and any custom styles.css rules must reference only these tokens.

| Role | Obsidian CSS Variable | 60/30/10 Mapping | Usage in Phase 1 |
|------|----------------------|------------------|------------------|
| Dominant surface | `--background-primary` | 60% | Settings tab background |
| Secondary surface | `--background-secondary` | 30% | Section headers, textarea background |
| Accent / interactive | `--interactive-accent` | 10% | Slider active track, focused toggle, active button |
| Accent text on accent | `--text-on-accent` | (paired with accent) | Text inside accent-colored controls |
| Normal text | `--text-normal` | — | All setting name labels |
| Muted / description text | `--text-muted` | — | Setting descriptions, help text, placeholder text |
| Error / destructive | `--text-error` | — | Inline validation messages for malformed exclusion patterns or folder rules |
| Input border | `--background-modifier-border` | — | Textarea and text input borders |
| Input border (focused) | `--interactive-accent` | — | Focused textarea / text input border |

Accent reserved for: slider thumb and active track, toggle-on state, focused input border.
Do not apply `--interactive-accent` to static text, section headings, or description copy.

## Settings Tab Structure

Phase 1 renders a single `PluginSettingTab` (`KBSettingsTab`) with three named sections.
This is the complete interaction surface for Phase 1.

### Section order in `display()`

```
[h3] General
  SET-01 — Update interval slider
  SET-04 — Auto-injection toggle

[h3] Exclusions
  SET-02 — Exclusion patterns textarea

[h3] MOC Format
  SET-03a — Default MOC format toggle (dedicated / inline)
  SET-03b — Per-folder rules textarea
```

### Section 1: General

**Setting: Update interval (minutes)**
- API: `new Setting(containerEl).setName(...).setDesc(...).addSlider(...)`
- Slider limits: min 1, max 60, step 1
- Default value: 5 (from D-05)
- Display pattern: slider with a live value readout via `.setDynamicTooltip()` or a
  companion `SliderComponent` tooltip — show current value as integer minutes
- `onChange`: write to `settings.updateIntervalMinutes`, call `saveSettings()`

**Setting: Auto-injection**
- API: `.addToggle(...)`
- Default: off / disabled (from D-07)
- `onChange`: write to `settings.autoInject`, call `saveSettings()`
- When off: plugin generates dedicated MOC files only
- When on: plugin will auto-inject inline sections (enforced in Phase 4+, toggle stored now)

### Section 2: Exclusions

**Setting: Exclusion patterns**
- API: `.addTextArea(...)`
- Rows: minimum 6 visible rows (approximate; use `setPlaceholder` not inline style)
- Default value: empty string (from D-06)
- Placeholder text: see Copywriting Contract below
- `onChange`: parse textarea value by splitting on newlines, trim each line, filter
  empty lines → `string[]` → write to `settings.excludedPaths`, call `saveSettings()`
- No validation UI required — malformed patterns are silently ignored per D-01/D-04

### Section 3: MOC Format

**Setting: Default MOC format**
- API: `.addDropdown(...)` with two options: `"dedicated"` (label: "Dedicated file") and
  `"inline"` (label: "Inline injection")
- Alternative acceptable: a toggle labeled "Use inline injection by default" if dropdown
  adds complexity — executor may choose either; both satisfy D-08
- Default: `"dedicated"` (from D-08)
- `onChange`: write to `settings.defaultMocFormat`, call `saveSettings()`

**Setting: Per-folder rules**
- API: `.addTextArea(...)`
- Rows: minimum 6 visible rows
- Default value: empty string (from D-08 — schema defined, no pre-seeded rules)
- Placeholder text: see Copywriting Contract below
- `onChange`: parse each non-empty line as `folder/path = dedicated|inline`; silently
  skip lines that don't match this pattern (from D-09); write parsed result to
  `settings.folderRules` as `Record<string, 'dedicated' | 'inline'>`, call `saveSettings()`

## Copywriting Contract

All copy below is the exact text to use. No paraphrasing in implementation.

### Setting Names and Descriptions

| Setting | `.setName()` | `.setDesc()` |
|---------|-------------|-------------|
| Update interval | `Update interval` | `How often KB Manager rebuilds MOC files and TOC sections in the background. Range: 1–60 minutes.` |
| Auto-injection | `Auto-injection` | `When enabled, automatically injects MOC sections into all notes in folders configured for inline format. Disabled by default — enable after reviewing per-folder rules.` |
| Exclusion patterns | `Exclusion patterns` | `Folders and files to skip entirely — no indexing, no MOC or TOC writes. One name per line. A pattern matches any path segment: "templates" excludes notes/templates/foo.md and templates/bar.md.` |
| Default MOC format | `Default MOC format` | `How MOC content is delivered when no per-folder rule applies. "Dedicated file" creates a MOC.md in each folder. "Inline injection" updates sections inside existing notes that contain delimiter markers.` |
| Per-folder rules | `Per-folder rules` | `Override the default MOC format for specific folders. One rule per line. Lines that don't match the expected format are ignored.` |

### Section Headings

| Section | `containerEl.createEl('h3', { text: '...' })` |
|---------|-------------------------------------------------|
| General | `General` |
| Exclusions | `Exclusions` |
| MOC Format | `MOC Format` |

### Placeholder Text

| Setting | Placeholder |
|---------|-------------|
| Exclusion patterns textarea | `templates\narchive\ndaily-notes` |
| Per-folder rules textarea | `notes/projects = inline\ndailies = dedicated` |

### Validation and Error States

Phase 1 does not surface validation errors in the UI — malformed lines are silently
ignored per D-01 and D-09. No error copy is required for Phase 1.

If the executor adds optional inline validation in a future revision, use this pattern:

| Condition | Message |
|-----------|---------|
| Exclusion line contains `/` at start | `Paths are matched as segments — no leading slash needed.` |
| Folder rule line malformed | `Expected format: folder/path = dedicated or folder/path = inline` |

### Destructive Actions

Phase 1 has no destructive actions. No confirmation dialogs required.

### Empty State

The exclusion patterns textarea and per-folder rules textarea show placeholder text when
empty (defined above). No additional empty state UI is needed — these controls are always
visible and editable.

## Interaction States

| Control | State | Visual Treatment |
|---------|-------|-----------------|
| Slider | Default | `--interactive-normal` track, thumb at current value |
| Slider | Active/dragging | `--interactive-accent` active track segment |
| Toggle | Off | `--interactive-normal` background |
| Toggle | On | `--interactive-accent` background |
| Textarea | Default | `--background-secondary` fill, `--background-modifier-border` border |
| Textarea | Focused | `--interactive-accent` border (Obsidian default focus ring) |
| Dropdown | Default | Obsidian native select styling |
| Dropdown | Open | Obsidian native dropdown open state |

All states are handled by Obsidian's built-in Setting component CSS. The plugin does not
need to write custom CSS for these states in Phase 1.

## styles.css Contract

Phase 1 requires minimal custom CSS. The `styles.css` file (if created) must only contain:

```css
/* KB Manager — Phase 1 */
/* No custom styles required for settings tab in Phase 1. */
/* Obsidian's Setting API handles all control styling via CSS variables. */
/* Add only if a layout issue cannot be resolved with the Setting API alone. */
```

Do not add CSS that overrides `--background-primary`, `--text-normal`, or any Obsidian
theme token. Theme compatibility is mandatory — the plugin must look correct in both
light and dark Obsidian themes without conditional CSS.
